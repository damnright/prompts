# iOS 微信小程序大尺寸图片滚动后白屏、闪退

> 知识类型：故障排查与性能治理  
> 适用范围：微信小程序、uni-app、原生 `<image>`、WebP/JPEG/PNG 大图  
> 关键词：iOS、图片解码、内存峰值、WebP、白屏、闪退、`getImageInfo:fail invalid`

## 问题摘要

微信小程序在 iOS 真机加载图片列表或富文本图片时，可能出现以下现象：

- 图片开始能够正常展示。
- 页面上下滚动、切换内容区域或反复进入页面后出现白屏。
- 随后小程序被强制退出，微信提示“小程序异常”。
- JavaScript 控制台没有对应异常堆栈。
- 某些 WebP 下载到本地后，调用 `wx.getImageInfo` 返回 `getImageInfo:fail invalid`。

这类问题通常不是单纯的“iOS 不支持 WebP”，而是以下两类问题可能同时存在：

1. 大像素图片被反复解码，造成原生位图、解码缓冲和 GPU 纹理的内存峰值，最终由系统终止小程序进程。
2. 图片响应头或下载后的临时文件缺少可靠类型信息，导致微信图片 API 无法识别 WebP 等格式。

## 核心结论

### 图片文件小，不等于展示内存小

图片文件大小是压缩后的网络传输体积。图片展示前必须解码成位图，常用估算公式为：

```text
理论位图内存 = 图片宽度 × 图片高度 × 4 bytes
```

例如，一张静态 WebP 的实测数据为：

| 项目 | 数值 |
| --- | ---: |
| 压缩文件大小 | 约 241 KiB |
| 像素尺寸 | 2500 × 1842 |
| 像素数量 | 4,605,000 |
| RGBA 理论位图内存 | 约 17.57 MiB |

实际峰值还可能包含：

- 图片格式解码缓冲区
- 原始位图和缩放副本
- GPU 纹理
- 原生图片缓存
- Canvas 绘制和导出缓冲

因此，单张图片的瞬时内存可能高于理论位图内存，多张图片并发或反复解码时风险会进一步放大。

### WebP 不是唯一原因

在相同像素尺寸下，WebP、JPEG 和 PNG 解码后的位图内存处于同一数量级。WebP 可以显著降低文件体积，但不能按相同比例降低解码后的位图内存。

不同平台和解码器的临时缓冲、缓存和格式兼容性可能存在差异，因此 WebP 可能更容易暴露问题，但不能仅凭文件格式断定是“WebP 解码器 Bug”。

### 延迟闪退更符合内存压力特征

一般可以按以下表现初步区分：

- 图片第一次加载就失败、空白或触发 `error`：优先排查格式、响应头、域名和文件损坏。
- 图片能够展示，滚动或切换数次后退出：优先排查重复解码、原生缓存和 GPU 内存压力。
- 进程直接退出且没有 JavaScript 错误：优先排查原生层崩溃或系统因内存压力终止进程。

以上只能用于快速定位，最终结论仍应结合 iOS 崩溃日志、微信性能面板和真机复现结果。

## `getImageInfo:fail invalid` 的常见原因

部分文件服务会把图片按普通附件返回，例如：

```http
Content-Type: application/octet-stream
Content-Disposition: attachment; filename=example.webp
X-Content-Type-Options: nosniff
```

这会产生以下风险：

- 下载后的临时文件可能没有正确扩展名。
- MIME 类型没有声明真实图片格式。
- `nosniff` 阻止客户端根据文件内容推断类型。
- `getImageInfo`、`compressImage` 或 Canvas 图片对象可能识别失败。

WebP 图片建议返回：

```http
Content-Type: image/webp
Content-Disposition: inline
```

JPEG 和 PNG 也应分别使用正确的 `image/jpeg`、`image/png`。

需要注意：修正响应头可以解决格式识别问题，但不能降低图片的解码内存。

## 常见但不充分的处理方式

### 只添加 `lazy-load`

`lazy-load` 只能推迟加载时机，不是严格的并发控制或内存上限，也不能保证已解码位图立即从原生缓存和 GPU 中释放。

### 只使用 `mode="widthFix"`

`widthFix` 改变的是页面显示尺寸，不一定改变原始图片的解码尺寸。把 2500px 宽的图片显示为 375px，仍可能先解码完整原图。

### 只隐藏非当前内容

使用 `opacity`、`visibility` 或移出屏幕隐藏组件，不能保证图片节点和已解码资源被释放。非当前内容应优先使用条件渲染卸载。

### 离开视口就卸载，返回后重新加载原图

这种方式能减少当前节点数量，但会反复触发大图解码。原生图片缓存和 GPU 资源未必同步释放，可能造成图片闪烁、空白和滚动数次后内存持续上升。

### 把 `application/octet-stream` 当作省内存方案

`application/octet-stream` 只是 HTTP 响应类型，不会改变图片的像素数量和解码成本，反而可能影响图片类型识别。

### 依赖客户端压缩彻底解决问题

客户端生成缩略图之前，仍需至少解码一次原图。如果单张原图的解码峰值已经超过设备承受范围，客户端压缩、Canvas 缩放和懒加载都无法保证安全。

## 推荐解决方案

### 方案一：服务端生成缩略图，优先推荐

这是最稳定的解决方案。小程序列表、富文本和介绍页面只加载缩略图，用户主动预览时才加载单张原图。

建议：

- 上传图片时异步生成多种尺寸。
- 页面展示图宽度控制在 750px 左右，最高通常不超过 1080px。
- 对超长图同时限制最大高度或最大像素数量。
- 返回正确的 `Content-Type` 和 `Content-Disposition: inline`。
- 缩略图使用稳定 URL 和长期缓存策略。
- 原图 URL 与缩略图 URL 分离，避免前端根据参数猜测处理结果。

可以按像素总量控制缩略图，而不仅仅限制文件体积：

```text
目标像素数量 = 目标宽度 × 目标高度
目标位图内存 ≈ 目标像素数量 × 4 bytes
```

### 方案二：客户端一次性生成并缓存缩略图，作为应急方案

当服务端暂时无法提供缩略图时，可以采用以下流程：

```text
远程原图
  -> 串行下载
  -> 保存为带真实扩展名的本地文件
  -> 尝试原生图片信息读取和压缩
  -> 不支持的格式使用 Canvas 2D 单张缩放
  -> 导出本地 JPEG/PNG 缩略图
  -> 删除本地原图文件
  -> 缓存缩略图路径供后续滚动复用
```

建议约束：

- 所有下载、解码、缩放和导出任务严格串行。
- 同时只处理一张原图。
- 限制缩略图最大宽度、最大高度和最大像素数量。
- 缩略图生成成功后立即释放 Canvas 和原图引用。
- 页面滚动返回时复用本地缩略图，不重新解码远程原图。
- 缩略图生成失败时显示可重试占位，不自动回退为大图内联展示。

例如，把 `2500 × 1842` 缩放为约 `750 × 552` 后，理论 RGBA 位图内存从约 17.57 MiB 降至约 1.58 MiB，下降约 91%。

客户端方案仍有一次原图解码峰值，因此只能作为过渡方案。如果首次生成缩略图就导致真机退出，必须改用服务端缩略图。

## 页面加载策略

在缩略图基础上，再使用页面级加载控制：

- 非当前 Tab 或折叠区域使用条件渲染完全卸载。
- 使用 `IntersectionObserver`，图片接近可视区时才加入队列。
- 首屏有多张图片时按文档顺序串行加载。
- 已生成的低分辨率缩略图可以在当前会话内复用。
- 页面隐藏或卸载时断开观察器、停止队列并清理引用。
- 收到 `wx.onMemoryWarning` 时降低驻留图片数量并暂停非必要 Canvas 工作。

不要把复杂的观察器、队列和缓存当作根本方案。它们只能管理加载时机，真正降低风险的关键仍然是减少图片解码像素。

## Canvas 与图片错峰

分享海报、截图和图片缩放都会使用 Canvas，可能与页面图片争夺内存。建议：

- 不在批量图片加载期间生成海报。
- 海报生成前暂停非必要图片加载。
- 导出后执行 `clearRect`，清除图片引用并卸载 Canvas。
- 新代码优先使用 Canvas 2D 接口。
- 避免多个 Canvas 同时处理大图。

## 原图预览策略

- 用户主动点击时再加载原图。
- 每次只预览当前一张原图。
- 不要一次把整个图片列表交给原生预览组件。
- 关闭预览触发页面生命周期时，避免重新刷新数据和重建全部图片。

## 验证清单

### 网络与加载行为

- 页面展示的是缩略图 URL 或本地缩略图路径，而不是远程大图。
- 同一张原图不会因反复滚动而重复下载、解码和转码。
- 首屏多图按顺序加载，不出现批量并发解码。
- 非当前内容区域的图片节点已经卸载。

### iOS 真机

- 反复切换包含图片的 Tab 或折叠区域。
- 连续上下滚动至少 10 次。
- 验证图片不会持续闪烁、空白或重新请求原图。
- 验证小程序不会白屏或被微信强制退出。
- 验证 Canvas 生成、单张原图预览和预览关闭后的页面恢复。
- 记录 `wx.onMemoryWarning`，但不能因为未收到警告就排除内存问题。

### 服务端

- 检查 `Content-Type` 是否与实际图片格式一致。
- 检查是否错误使用 `attachment` 或 `application/octet-stream`。
- 检查缩略图的真实像素尺寸，而不仅是文件大小。
- 检查 CDN 或对象存储是否正确缓存缩略图。

## 快速判断表

| 现象 | 优先判断 | 处理方向 |
| --- | --- | --- |
| `getImageInfo:fail invalid` | MIME、扩展名、临时文件或格式识别失败 | 修正响应头并保留真实扩展名 |
| 图片能展示，滚动数次后退出 | 重复解码、原生缓存或 GPU 内存压力 | 使用并复用低像素缩略图 |
| 单张图第一次加载就退出 | 单图解码峰值过高或平台解码异常 | 服务端生成缩略图 |
| 文件很小但仍闪退 | 压缩体积小但像素尺寸大 | 按 `宽 × 高 × 4` 估算内存 |
| `lazy-load/widthFix` 后仍闪退 | 只改变时机或显示尺寸 | 加载前真正生成小尺寸文件 |
| 图片频繁闪烁或空白 | 视口进出时反复卸载和解码 | 缓存并复用低分辨率缩略图 |

## 可复用经验

1. 移动端图片问题应优先检查像素尺寸和解码内存，不能只看文件大小。
2. “无 JavaScript 错误、操作数次后异常退出”应优先排查原生内存压力。
3. 懒加载、条件渲染和串行队列是辅助措施，真正的内存治理依赖低像素缩略图。
4. 频繁重新绑定原图可能比长期复用低分辨率缩略图更危险。
5. Canvas 与页面图片应错峰工作，使用后及时清空和卸载。
6. 正确的图片响应头解决格式识别问题，但不能代替缩略图。
7. 服务端缩略图比客户端首次解码原图更安全，是大图场景的最终方案。
